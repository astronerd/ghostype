LangChain 把上下文分成三层（来源）：

Runtime Context（静态配置）— 用户 ID、权限、环境变量
State（短期记忆）— 当前对话消息、工具结果
Store（长期记忆）— 用户偏好、历史数据
它用 middleware + 声明式的方式，让 agent 按需从不同数据源拉取上下文，而不是 hardcode。system prompt 甚至可以是动态的——根据对话状态、用户偏好从 store 里读取后拼装。

对应到 GHOSTYPE 的 skill 系统，问题就很清楚了：

现在 SkillExecutor 里 hardcode ghost_profile 就像是把所有 context 写死在代码里，而不是让 skill 声明自己需要什么
更好的做法是 skill 声明需要的 context 变量（比如 context_requires: [ghost_profile, calibration_records]），SkillExecutor 有一个 provider 注册表，按需去对应的 store 取数据
system prompt 里用 {{context.xxx}} 模板变量注入高层策略相关的上下文（比如 profile 的结构说明、出题策略）
user message 里传每次变化的具体数据（比如"请出题"、具体的历史记录）
